# ADR-002 — Архитектура оркестратора

**Статус:** принято  
**Дата:** 2025-04

---

## Контекст

Система должна принимать текст на естественном языке и определять, какой из девяти воркфлоу следует выполнить. Требования к классификатору:

- Точность: студенческие запросы могут быть неоднозначными, жаргонными, с опечатками
- Скорость: пользователь не должен ждать классификации дольше ~2 секунд
- Отказоустойчивость: недоступность OpenAI не должна останавливать систему полностью
- Расширяемость: добавление нового воркфлоу не должно требовать правок в роутере или классификаторе

---

## Решение

**Гибридный классификатор LLM-first + keyword fallback, работающие параллельно.**

### Архитектура оркестратора

```
orchestrate(input)
  ├── classifyLLM(text)         — GPT-4o, json_object, timeout 8 с
  ├── classifyByRegistry(text)  — keyword scan O(n·k), всегда < 5 мс
  │
  └── Выбор победителя:
        LLM confidence ≥ 0.75  → использовать LLM-результат
        Keyword conf ≥ 0.75    → использовать keyword-результат
        Оба < 0.75             → использовать результат с более высоким score
        LLM timeout/ошибка     → использовать keyword-результат

Политика confidence:
  ≥ 0.75  → executeWorkflow() напрямую
  0.45–0.74 → вернуть needsClarification = true
  < 0.45  → buildFallbackResponse()
```

### Принцип единого реестра

Все воркфлоу описаны в одном файле — `lib/orchestrator/registry.ts`. Каждая запись содержит:
- уникальный идентификатор (`WorkflowName`)
- ключевые слова для быстрого matching
- минимальный порог confidence
- функцию `run(text, ctx)` — единственная точка входа в бизнес-логику

**Добавление нового воркфлоу = одна запись в registry.ts.** Router, executor и классификатор не меняются.

## Рассмотренные альтернативы

### Чисто rule-based (только ключевые слова)
**Отклонено для финального решения.** Не справляется с перефразировками («помоги с заявой» вместо «напиши заявление»), синонимами и смешанными запросами. Достаточен только как fallback.

### Чисто LLM-классификатор
**Отклонено.** Недоступность OpenAI → полная деградация системы. Latency 1–3 секунды на каждый запрос даже для простых случаев, которые keyword matching решает за <5 мс.

### Отдельный fine-tuned классификатор (BERT / distilbert)
**Отклонено.** Требует разметки обучающей выборки, хостинга модели, деплой-пайплайна. Избыточно для MVP. Может быть рассмотрено в Phase 3 при накоплении датасета из `orchestrator_runs`.

### Embeddings-based классификатор (cosine similarity по примерам)
**Отложено.** Интересный подход: хранить эталонные запросы для каждого воркфлоу, классифицировать по ближайшему вектору. Требует кураторства примеров. Может заменить keyword matching в Phase 2.

### n8n / Zapier для оркестрации
**Отклонено.** Внешний сервис = сетевой hop на каждый запрос. Сложно типизировать. Код оркестрации должен быть в репозитории и под контролем версий.

## Последствия

### Положительные
- Параллельный запуск LLM + keywords: итоговая latency = max(LLM, KW) ≈ latency LLM, но с готовым fallback
- Система работает при недоступном OpenAI (деградированное качество, но не падение)
- Все пороги confidence вынесены в `policies.ts` — настройка без изменения логики
- Журнал каждого решения в `orchestrator_runs` позволяет анализировать качество классификации

### Отрицательные и риски
- LLM-классификатор стоит денег (~$0.00015 за запрос с GPT-4o-mini) — не критично на MVP, но важно отслеживать при масштабировании
- При таймауте 8 секунд возможна деградация UX если OpenAI медленно отвечает — решено параллельным запуском keyword matching
- Пороги confidence требуют калибровки на реальных данных; текущие значения (0.45/0.75) — начальные эвристики
