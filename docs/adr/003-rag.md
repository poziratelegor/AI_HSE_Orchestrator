# ADR-003 — Стратегия RAG-пайплайна

**Статус:** принято  
**Дата:** 2025-04

---

## Контекст

Воркфлоу `rag_qa` должен отвечать на вопросы по документам пользователя: PDF-конспектам, TXT-материалам и аудиозаписям лекций. Требования:

- Точность: ответы должны быть строго обоснованы найденными фрагментами, без галлюцинаций
- Прозрачность: пользователь должен видеть, откуда взят каждый тезис (цитаты с указанием документа)
- Скорость: P95 ≤ 6 секунд от запроса до ответа
- Мультимодальность: поддержка PDF, TXT и аудио (MP3, WAV, OGG, WebM)
- Изоляция: поиск только по документам текущего пользователя (RLS)

---

## Решение

**Supabase Storage + pgvector + text-embedding-3-small + query expansion.**

### Пайплайн загрузки

```
POST /api/upload
  → проверка MIME + magic bytes
  → INSERT documents (status: pending)
  → ответ за < 500 мс

[Background: waitUntil()]
  → извлечение текста:
      PDF     → pdf-parse
      Аудио   → OpenAI Whisper (транскрипция)
      TXT     → UTF-8 чтение
  → chunkText(text, 800 токенов, overlap 2 предложения)
  → embedBatchSafe(chunks, модель: text-embedding-3-small)
  → INSERT document_chunks (id, document_id, chunk_index, chunk_text, embedding)
  → UPDATE documents (status: ready | partial | failed)
```

### Пайплайн запроса

```
POST /api/rag/query
  → Redis cache-first (TTL 1 час, ключ: rag:{userId}:{sha256(query)})
  → expandQuery(query) → 3 варианта перефразировки через GPT-4o-mini
  → embedText(каждый вариант) → vector[]
  → match_document_chunks(vector, threshold=0.5, k=5, user_id)
  → guardContext(chunks) → обрезать до context window
  → buildCitations(chunks) → выдержки 200 символов
  → GPT-4o-mini: "Ответь строго по этим фрагментам: ..."
  → Redis SET result
  → вернуть { answer, citations }
```

### Стратегия чанкинга

Разбивка по предложениям и абзацам с мягкими границами:
- Размер чанка: 800 токенов (~3200 символов)
- Перекрытие: 2 последних предложения предыдущего чанка переносятся в начало следующего
- Обнаружение заголовков: Markdown (`#`), жирный текст, ALL CAPS — используются как естественные точки разбивки
- Минимальный чанк: 100 токенов (более короткие сливаются с предыдущим)

## Рассмотренные альтернативы

### LangChain / LlamaIndex вместо собственного пайплайна
**Отклонено.** Добавляет зависимость от быстро меняющегося фреймворка с нестабильным API. Собственная реализация из 5 файлов (`chunk.ts`, `embed.ts`, `retrieve.ts`, `expand-query.ts`, `citations.ts`) полностью прозрачна, тестируема и не приносит лишних абстракций.

### Фиксированный размер чанка (split by chars)
**Отклонено.** Разрезание по символам разрывает предложения посередине, что снижает качество эмбеддинга. Семантически цельный фрагмент векторизуется точнее.

### Только один вариант запроса (без query expansion)
**Отклонено.** Студенческие вопросы часто неточны или используют другую терминологию, чем в документе. Expansion до 3 вариантов значительно улучшает recall без существенного роста стоимости (~$0.0001 за расширение).

### Pinecone / Qdrant для векторного хранения
**Отклонено.** pgvector в составе Supabase достаточен для объёмов MVP. Переход на специализированное решение оправдан при > 5M векторов или требовании sub-10ms поиска — на этом этапе не нужен.

### Хранение файлов в Supabase Storage и повторный парсинг при каждом запросе
**Отклонено.** Парсинг PDF и транскрипция аудио — дорогие операции. Единовременная обработка при загрузке и хранение чанков в БД — правильная стратегия.

### Без кэширования (Redis)
**Отклонено.** Повторные запросы с одинаковым текстом (студент переспрашивает) стоят денег и добавляют latency. Redis кэш с TTL 1 час даёт мгновенный ответ и экономит ~40% стоимости запросов при типичных паттернах использования.

## Последствия

### Положительные
- Ответы строго ограничены найденными фрагментами → минимум галлюцинаций
- Цитаты с указанием документа и excerpt → пользователь может проверить источник
- Кэш RAG-ответов сокращает latency до ~50 мс на повторных запросах
- `content_hash` на таблице `documents` предотвращает повторный инжест одного файла

### Отрицательные и риски
- Качество retrieval напрямую зависит от качества чанкинга — плохая разбивка = плохие ответы
- Whisper транскрипция стоит ~$0.006/мин, что критично для длинных лекций (2 часа = ~$0.72)
- Аудио > 25 МБ не поддерживается Whisper API — нужна предварительная нарезка для длинных записей
- pgvector HNSW-индекс требует `VACUUM` при интенсивных вставках для поддержания производительности
